# Create Your First Application

Thunder is a lightweight identity platform for authentication, authorization, and user management. It supports standard protocols like OAuth 2.0 and OIDC, flexible sign-in flows, and individual mechanisms such as password, passwordless, and social login. This guide walks you through registering your first application in Thunder so you can start integrating sign-in into your app.

:::tip Connect with AI
Install the [Thunder MCP server](/guides/working-with-ai/mcp-server) to manage Thunder directly from your AI assistant. Create applications, configure flows, and get integration code without leaving your IDE.
:::

## Prerequisites

- Thunder is running locally or on a server. See [Get Thunder](/guides/getting-started/get-thunder) to get started.
- You have access to the Thunder Console or a system API token. For API access, obtain a system token from the Thunder Console.

## Create an Application

An application in Thunder defines how your users authenticate and what they can access. Register it first to get a **Client ID** for your integration.

<Tabs>
<TabItem value="console" label="🖥️ Console">

1. Sign into the Thunder Console at [https://localhost:8090/develop](https://localhost:8090/develop).
2. Navigate to **Applications** → **New Application**.
3. Enter an **Application Name**.
4. Select the **Sign-In Options** — the methods users will use to sign in (for example, Username & Password or social login).
5. Configure the **Sign-In Experience** — select how and who can access your application:

   - **Sign-In Approach**: Choose how users access the sign-in flow.
     - **Redirect to Thunder sign-in/sign-up handling pages** (recommended) — Thunder redirects users to system-hosted pages, which you can customize with the Flow Designer and integrate with SDKs in a few steps.
     - **Embedded sign-in/sign-up components in your app** — users sign in directly through your app using Thunder's UI components or APIs.

   - **User Access**: Select which user types can access this application — **Customer**, **Person**, or both.

      <details>
        <summary>User Types</summary>

        User types define the profile and attributes for different categories of users.

        Thunder includes the following default user types:

        - **Customer**: End users who interact with your application. This type allows self-registration.
        - **Person**: Administrator-managed general-purpose user accounts.

      </details>

6. Select the **Application Type** and provide a redirect URL if prompted.

    <details>
      <summary>Application Types</summary>

      | Application Type       | Description                                                                              | Required Configuration                                                                                                                                                                  |
      | ---------------------- | ---------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
      | ⚛️ **Browser App**     | For browser-based single-page applications (React, Angular, Vue).                       | Configure an **Authorized Redirect URL** (for example, `http://localhost:3000/callback`) to receive the authentication response.                                                        |
      | 🌐 **Full-stack App**  | For server-rendered web applications (Express, Spring, .NET).                           | Configure an **Authorized Redirect URL** where the server handles the authentication response.                                                                                          |
      | 📱 **Mobile App**      | For native or cross-platform mobile applications (Swift, Kotlin, React Native, Flutter). | Configure a **Deep Link** (for example, `com.example.app://auth/callback`) or **Universal Link** (for example, `https://app.example.com/auth/callback`) to receive the sign-in response. |
      | 🤖 **Backend Service** | For server-to-server communication without user interaction.                             | No redirect URL required. Use the generated **Client ID** and **Client Secret** for authentication.                                                                                     |

    </details>
7. Click **Create Application**.

:::tip
Note the **Client ID** from the application details page. You need it to configure your integration.
:::

</TabItem>
<TabItem value="api" label="🧩 API">

```bash
curl -kL -X POST https://localhost:8090/applications \
  -H 'Authorization: Bearer <your-system-token>' \
  -H 'Content-Type: application/json' \
  -d '{
    "name": "my-app",
    "inbound_auth_config": [{
      "type": "oauth2",
      "config": {
        "grant_types": ["authorization_code", "refresh_token"],
        "response_types": ["code"],
        "redirect_uris": ["http://localhost:5173"],
        "token_endpoint_auth_method": "none",
        "public_client": true,
        "pkce_required": true
      }
    }]
  }'
```

:::warning Important
Note the **Client ID** from the response. You need it to configure your integration.
:::
